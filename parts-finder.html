01 · Customer problem

Parts search begins with incomplete language.

Customers may provide a vehicle, year, model, or part across multiple messages and expect the system to remember context.

02 · System response

Conversation becomes structured fitment state.

LangGraph extracts and validates vehicle details while the session layer preserves only the temporary context needed for the search.

03 · Customer value

The interface guides rather than exposes the architecture.

Text, voice, progress states, and product cards create one focused workflow designed for customer-facing tablet and desktop use.

Architecture · 03

One customer workflow, several isolated system responsibilities.

React owns the customer interaction. FastAPI defines the service boundary. LangGraph manages state transitions, model routing extracts vehicle data, and MCP keeps product access separate from conversational reasoning.

Engineering decisions · 04

Reliability and privacy are part of the product behavior.

The implementation does more than connect an interface to a model. It validates extraction quality, recovers from provider failures, limits temporary data, and communicates voice processing honestly.

Model routing

Local-first extraction with cloud fallback

Ollama is the primary extraction provider. Network failures, malformed JSON, incomplete explicit-year extraction, and refusal responses automatically route to Groq behind the same typed result contract.

Voice

Controlled transcription flow

The browser records a short audio clip and sends it to a dedicated FastAPI endpoint backed by Groq Whisper, avoiding dependence on the browser's unreliable speech service.

Orchestration

Explicit LangGraph state

Vehicle details, intent, next-step routing, replies, and products move through a defined graph instead of being hidden in one large prompt.

Tool boundary

MCP-isolated product access

The backend starts an MCP server over stdio and invokes a typed product-search tool, keeping product retrieval independent from the agent workflow.

Privacy

Temporary context, not customer profiles

Sessions live only in server memory, expire after five minutes, can be cleared immediately, and do not write customer vehicle fields to application logs.

Customer experience

A premium interface with a narrow decision path

  • Responsive React layout designed for customer-facing tablets and desktops
  • Clear empty, listening, transcribing, loading, and product-result states
  • Plain-language privacy disclosure for temporary search and voice processing
  • One focused action: describe the vehicle and find compatible options
Verification

Automated checks across the stack

  • 24 backend tests covering graph, routes, fallback, nodes, and expiration
  • Frontend ESLint validation
  • Production Vite build verification
  • Typed FastAPI request and response models
Current boundary

Product data is intentionally mocked

The MCP tool currently returns representative products. A production release would replace that function with the real Bloomreach catalog, price, inventory, and fitment APIs.

Engineering learnings

What this implementation reinforced

  • Provider fallback must include response-quality failures, not only exceptions
  • Customer-facing AI benefits from progressive context collection rather than one perfect prompt
  • Privacy statements should describe actual storage, logging, and third-party processing behavior
  • Tool protocols such as MCP keep product integrations replaceable as the agent evolves
